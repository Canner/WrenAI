# CONTEXT

Ubiquitous language for the Wren project. Domain-meaningful terms only — not implementation detail.

## Data apps (feature: agent-generated Streamlit apps)

Command namespace: **`wren genbi`** — a Typer subcommand group under the `wren` CLI (not a separate binary), registered conditionally behind the `[streamlit]` extra (mirroring the `memory` block in `cli.py`). Verbs: `serve <name>`, `list`, `create`/`scaffold <name>`, `check <name>`, `stop <name>`, `logs <name>`.

- **Data app** — a Streamlit application generated by an agent and served locally by Wren, living at `<project>/apps/<name>/` as a first-class, version-controllable project artifact (parallel to `models/`, `views/`, `cubes/`). The agent hands the user a `http://localhost:<port>` URL. Decided per-conversation: ranges from a single static chart to a multi-dimension interactive dashboard.

- **Panel** — one chart/table/metric within a data app, backed by exactly one query.

- **Cube-expressible (panel)** — a panel whose query can be expressed as a cube query (measures + dimensions + filters over a single cube). This is the decision axis for panel architecture, NOT "interactive vs static":
  - *Cube-expressible* → bind the panel to a **cube query spec**; widgets mutate the spec (`dimensions`/`filters`/`timeDimensions`) and `cube_query_to_sql()` regenerates governed SQL each rerun. Default path. Governed by construction; the reachable query space equals the set of specs the cube allows, so it is finite and validated at build time. Adding interactivity later is free.
  - *Not cube-expressible* (complex window functions, cross-cube joins, row-level detail) → **raw SQL**, validated once at build time via `wren app check` (dry-plan), served **static** (no runtime structural variation).

- **Interactive panel** — a cube-expressible panel with Streamlit widgets bound to its cube spec. Interactivity is a runtime concern (whether widgets are attached), independent of the cube-expressible/raw axis.

- **App catalog** — the persistent, committable source of truth for *which apps exist*: the `apps/` folder + a single **`apps/index.yml`** mapping app name → entry file + metadata (description, bound cube). Each app's implementation lives in `apps/<name>/`. The index is committed to git; ephemeral serve state (`apps/.run/`) is gitignored. The agent reads the catalog to answer "what apps do I have" (e.g. after a reboot) and then serves a chosen one. (Trade-off accepted: a single index can merge-conflict in multi-author projects, but apps live in the user's own wren project, where this is rare.)

- **Serve runtime** — the *ephemeral, disposable* state of which app is running on which port. Not durably managed: a reboot ends all serving processes, and the user/agent simply re-serves on demand. Lifecycle is intentionally NOT supervised in v1.

- **MDL drift** — a committed app whose declared cube/dimensions no longer match the evolved MDL. Handled proactively: `wren genbi serve` auto-runs `check` first and refuses to serve a drifted app (naming the missing dimension/cube), and `wren context build` warns "N apps may be affected — run `wren genbi check --all`." Never a silent runtime traceback.

- **Governance boundary** — every reachable query must pass through the semantic layer (`WrenEngine` / `cube_query_to_sql`), so `strict_mode` / `denied_functions` apply. Raw SQL string concatenation in `app.py` is forbidden; runtime variation happens via the cube spec (a closed vocabulary), never by interpolating widget values into SQL text.

- **Panel helper (`wren_genbi.cube_panel`)** — a runtime helper module shipped with the `[streamlit]` extra. The agent writes **freeform** Streamlit layout/widgets, but every cube panel's data path goes through the helper (`cube spec → cube_query_to_sql → WrenEngine.query → cache → render`). The agent never hand-rolls SQL or the governance path. Consequences: governance is structural (not convention), and because each panel *declares* its cube + dimensions, MDL-drift detection (`wren genbi check`) is a trivial spec-vs-MDL comparison. Raw `engine.query(sql)` remains the documented escape hatch for non-cube / complex panels (validated once at build time, served static). The helper also owns the freshness UX: a tunable-`ttl` cache, a visible "last updated" timestamp, and a manual refresh control that clears the cache — so the user always knows how fresh the data is and can pull a fresh result.

### Dependencies / prerequisites
- **pip dependency** — `wrenai[streamlit]` extra (the runtime + Streamlit). Installable on demand.
- **data prerequisite** — interactive/cube-expressible panels require **cubes to exist** in the project. Not installable; if absent, route the user to cube creation (`wren-enrich-context` cube flow) before building an interactive app.
