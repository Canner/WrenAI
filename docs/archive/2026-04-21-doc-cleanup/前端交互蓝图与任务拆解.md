## 3. 已确认的产品决策

## 3.1 一级导航
一级导航固定为：

1. **新对话**
2. **我的知识库**
3. **工作空间**
4. **数据看板**
5. **设置**（左下角）

### 不再保留为主概念
- 团队知识库
- 知识花园

说明：
- 系统的真实隔离边界是 **workspace**，不是“团队知识库”。
- “知识花园”若未来需要保留，其能力应并入“工作空间中心 / 工作空间发现”，而不是继续作为一级主导航。

## 3.2 领域对象关系

### Workspace
- 类似租户 / 业务空间 / 数据隔离边界
- 用户可加入多个 workspace
- 用户当前始终工作在一个 workspace 内
- 当前 workspace 需要高频切换能力

### Knowledge Base
- 归属于某个 workspace
- 一个 workspace 下可有多个知识库
- “我的知识库”展示用户有权限看到的知识库

### Thread
- 归属于某个 workspace
- **Thread 绑定的不是单个 project，而是 `workspace + knowledgeBaseIds[]`**
- 支持在“新对话”中选择多个知识库作为问答范围
- thread 创建后，不允许在同一个 thread 中动态修改知识库范围

### Dashboard
- 归属于某个 workspace
- 一个 workspace 下可有多个 dashboard
- 可从问答结果沉淀，也可独立查看与编辑

## 4. 全局交互原则

## 4.1 页面单一主任务
- 新对话页：发起一次新的问答
- 我的知识库：浏览与管理知识库
- 工作空间：切换 / 发现 / 申请 / 管理 workspace
- 数据看板：浏览当前 workspace 下的 dashboard
- 设置：个人与系统偏好设置

## 4.2 Workspace 是第一上下文
- 知识库、对话、看板全部以当前 workspace 为默认上下文
- 右上角提供 workspace 高频切换器
- 页面内部默认只操作“当前 workspace”的对象，减少跨空间混乱

## 4.3 所有点击必须有结果
点击后只允许以下结果之一：
- 跳页
- 打开弹窗 / 抽屉
- 触发操作并进入 loading
- disabled 并展示原因

禁止：
- 可点击但仅弹“后续开放” toast
- 视觉上像按钮但没有 handler
- “假入口”长期暴露在主路径上

## 4.4 主 / 次操作分级
每个页面只保留一个主按钮：
- 新对话：发送问题
- 我的知识库：创建知识库
- 知识库详情空态：添加资产
- 规则页：新增规则
- SQL 模板页：新增模板
- 数据看板：新建看板

## 4.5 范围锁定规则
### 对话前
- 用户可选择一个或多个知识库
- 选择集合必须属于同一个 workspace

### 对话后
- thread scope 固化为：
  - `workspaceId`
  - `knowledgeBaseIds[]`
- 若用户要改知识库范围，只能重新新建对话

Workspace 切换器不承担：
- 申请加入新 workspace
- 完整权限管理
- 成员审批

## 6. 页面级交互蓝图

## 6.1 新对话

### 页面目标
在当前 workspace 下，基于一个或多个知识库发起新的问答。

### 页面结构
#### 顶部区域
- 欢迎语 / 产品说明
- 当前 workspace 标识

#### Composer 上方：范围选择区
- 当前 workspace
- 已选知识库 chips
- “选择知识库”按钮
- 模式切换（标准 / 技能）
- 技能入口

#### Composer 区
- 多行输入框
- 发送按钮
- 输入提示

#### 下方内容
- 推荐问题卡片
- 最近常用知识库 / 最近常用组合（可选）

### 关键交互
#### 选择知识库
- 点击弹出知识库选择器
- 支持同 workspace 内多选
- 支持搜索
- 选中后以 chips 形式回填在 composer 上方

#### 技能入口
- 点击弹出技能选择器（参考 `12 对话-技能选择.jpg`）
- 分类 + 搜索 + 卡片列表
- 确认后绑定到当前新对话输入上下文

### 创建 thread 行为
- 点击发送后，创建 thread
- thread 绑定：`workspaceId + knowledgeBaseIds[] + selectedSkills[]`
- 成功后跳转 `/home/[id]`

│                                              │

## 7. Workspace 可见性与加入机制建议

## 7.1 结论
**采用“工作空间中心”方案，不放设置，不再叫知识花园。**

## 7.2 原因
### 不放设置
- 频率高
- 是业务上下文切换，不是个人偏好
- 放设置太深

### 不继续叫知识花园
- 与真正的租户 / 隔离逻辑不匹配
- 容易与“我的知识库”混淆
- 容易误导成内容广场，而非权限空间

## 7.3 最终方案
### 高频切换
- 右上角 workspace 切换器

### 管理 / 申请 / 发现
- 一级导航“工作空间”中完成
