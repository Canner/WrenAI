# 数据测试造数方案（面向已补的 11 个 SQL 模板）

## 1. 目的

在 `docs/业务需求/knowledge-base/` 已经补出 11 个 `draft_sql` 模板后，下一步建议先做一套**最小可验证测试数据**，用于：

1. 校验 SQL 是否能正常跑通；
2. 校验业务口径是否符合文档；
3. 给后续知识库导入后的联调留一套固定回归样例。

---

## 2. 本轮建议覆盖范围

### 已可测试的 SQL 模板（11 个）

- T01 渠道日基础汇总
- T02 渠道与折扣映射
- T03 首存 cohort 提取
- T04 cohort 累计收入
- T06 TOP3/非TOP3 分层
- T08 首存 cohort 续存
- T09 所有用户区间汇总
- T10 首存用户日龄趋势
- T11 按游戏类型分布
- T12 TOP3/5 游戏类型分层
- T13 首存金额分桶

### 暂不纳入造数范围的模板（4 个）

- T05 cohort ROI：缺投放金额
- T07 VIP 最高等级分层：缺“统计区间最高 VIP”SQL 化模型
- T14 投放金额并表：缺投放金额
- T15 流量指标并表：缺 PV / UV / 下载点击UV

---

## 3. 推荐测试窗口

建议先固定一套较短窗口，便于人工验算：

- 综合统计区间：`2026-04-01 ~ 2026-04-07`
- cohort 区间：`2026-04-01 ~ 2026-04-03`
- `top_n = 3`
- `n_days = 7`
- `period_days = 7`

这样可以同时覆盖：
- 日报类按天统计
- 首存 cohort
- 2~6 存
- TOP3 / 非TOP3
- 游戏类型分布
- 首存金额分桶

---

## 4. 推荐最小造数规模

建议先做 **1 个平台 + 2 个渠道 + 7 个玩家 + 7 天行为数据**。

### 4.1 维表/基础表

至少准备：

- `tenant_plat`：1 条启用平台数据
- `channel`：2 条渠道数据
  - 主测试渠道：`channel_id = 1001`
  - 对照渠道：`channel_id = 1002`
- `report_channel_data_percent_config`
  - 给 `1001` 配折扣，例如 `90%`
  - 不给 `1002` 配折扣，用来验证默认 `100%`
- `game_type`：至少 3 个类型
  - 电子
  - 体育
  - 棋牌
- `game_line_series`：至少 1 条
- `game_line`：至少 3 条，对应 3 个游戏类型

---

## 5. 玩家场景设计

建议按下面的最小用户集造数。

| 玩家 | 渠道 | 注册日 | 首存日 | 首存金额 | 充值次数 | 排名角色 | 主要验证点 |
| --- | --- | --- | --- | ---: | ---: | --- | --- |
| P01 | 1001 | 04-01 | 04-01 | 10 | 6 | TOP1 | 新客首存、2~6存、TOP1、累计收入、营销/洗码/任务/优惠 |
| P02 | 1001 | 03-31 | 04-01 | 20 | 3 | TOP2 | 开发人数、非当日注册首存、累计收入 |
| P03 | 1001 | 04-02 | 04-02 | 50 | 2 | TOP3 | TOP3 边界、游戏类型分布 |
| P04 | 1001 | 04-02 | 04-02 | 2000 | 1 | 非TOP | 非TOP 用户、2000 档 |
| P05 | 1001 | 04-03 | 04-03 | 88 | 1 | 非TOP | 首存金额“其他”档 |
| P06 | 1001 | 04-01 | 无 | 无 | 0 | 无 | 仅注册/登录，无首存 |
| P07 | 1002 | 04-01 | 04-01 | 100 | 2 | 对照渠道 | 验证渠道过滤隔离 |

## 6. 各事实表建议造什么数据

## 6.1 `dim_player`

至少保证：

- `P01/P03/P04/P05/P06` 注册在主渠道 `1001`
- `P02` 注册也在 `1001`，但注册时间早于测试窗口，用于验证“开发人数”
- `P07` 在 `1002`
- 给部分玩家预置不同 `vip_id`，但当前**不依赖** VIP 历史口径

## 6.2 `dwd_player_login_log`

至少造：

- `P01 ~ P06` 在 `04-01 ~ 04-03` 各有若干登录
- `P07` 在 `1002` 有登录
- 至少留 1 个只有登录、没有充值的用户（`P06`）

用于验证：
- T01 登录人数
- 渠道过滤

## 6.3 `dwd_order_deposit`

建议最关键，按下面思路造：

### 主渠道 `1001`

- `P01`
  - 04-01：1充 = 10
  - 04-02：2充 = 30
  - 04-03：3充 = 50
  - 04-04：4充 = 100
  - 04-05：5充 = 200
  - 04-06：6充 = 300
- `P02`
  - 04-01：1充 = 20
  - 04-03：2充 = 100
  - 04-05：3充 = 200
- `P03`
  - 04-02：1充 = 50
  - 04-06：2充 = 100
- `P04`
  - 04-02：1充 = 2000
- `P05`
  - 04-03：1充 = 88

### 对照渠道 `1002`

- `P07`
  - 04-01：1充 = 100
  - 04-02：2充 = 100

用于验证：
- T01 存款人数/金额、首存人数、首存金额、新客首存、开发人数
- T03 首存 cohort 提取
- T08 2~6 存人数/率/人均金额
- T10 日龄趋势
- T13 首存分桶（10 / 20 / 50 / 2000 / 其他 / 100）

## 6.4 `dwd_order_withdrawal`

建议主渠道至少造：

- `P01`：04-03 提款 40
- `P02`：04-04 提款 20
- `P04`：04-05 提款 100

对照渠道：
- `P07`：04-03 提款 10

用于验证：
- T01 提款人数/金额、充提差
- T09 / T10 投充比与杀率相关区间汇总

## 6.5 `dwd_bet_order`

建议保证主渠道总有效投注排名明确：

- `P01`：累计有效投注 ≈ 3000（TOP1）
- `P02`：累计有效投注 ≈ 2500（TOP2）
- `P03`：累计有效投注 ≈ 1500（TOP3）
- `P04`：累计有效投注 ≈ 200（非TOP）
- `P05`：累计有效投注 ≈ 100（非TOP）
- `P06`：0
- `P07`：对照渠道单独造，不参与主渠道 TOP3

同时给每个玩家分配不同 `win_loss_amount`，例如：

- `P01`：+300
- `P02`：+180
- `P03`：+120
- `P04`：-30
- `P05`：+10

用于验证：
- T01 投注人数 / 有效投注 / 输赢
- T04 累计渠道收入中的输赢项
- T06 TOP3/非TOP3 分层
- T09 区间汇总
- T10 日龄趋势

## 6.6 `bds_bet_order_detail`

建议把下注再拆到至少 3 个游戏类型：

- 游戏类型 A：电子
- 游戏类型 B：体育
- 游戏类型 C：棋牌

分布示例：

- `P01`：电子 + 体育
- `P02`：棋牌为主
- `P03`：体育为主
- `P04`：电子
- `P05`：电子少量
- `P07`：对照渠道独立分布

用于验证：
- T11 游戏类型分布
- T12 TOP3 / 非TOP3 游戏类型分层

## 6.7 收入抵扣/成本相关表

这些表建议至少各造 1~2 条，最好都落在主渠道 `1001`：

- `dwd_order_rebate`
  - `P01`：04-02 金额 5
  - `P02`：04-03 金额 10
- `dwd_order_task`
  - `P01`：04-03 金额 8
- `dwd_order_activity`
  - `P01`：04-02 金额 20
- `dwd_order_promote_activity`
  - `P02`：04-04 金额 15
- `dwd_order_add_or_sub`
  - `P01`：04-03 `1207` 金额 12（优惠加）
  - `P02`：04-05 `2204` 金额 6（优惠减/逆向场景）

用于验证：
- T01 洗码/活动/任务/优惠
- T04 渠道收入 = 输赢 - 洗码 - 任务 - 营销 - 优惠加扣款

## 6.8 仅 T01 需要补的表

- `dwd_order_lottery`
  - `P01`：04-02 金额 10
- `dwd_order_vip_award`
  - `P01`：04-02 金额 30

用于验证：
- T01 营销/彩金类字段是否被正确汇总

## 7. 建议造数顺序

建议按下面顺序执行，最省事：

1. 维表：`tenant_plat` / `channel` / `report_channel_data_percent_config`
2. 游戏维表：`game_type` / `game_line_series` / `game_line`
3. 玩家：`dim_player`
4. 行为主表：
   - `dwd_player_login_log`
   - `dwd_order_deposit`
   - `dwd_order_withdrawal`
   - `dwd_bet_order`
   - `bds_bet_order_detail`
5. 收入抵扣/彩金类表：
   - `dwd_order_rebate`
   - `dwd_order_task`
   - `dwd_order_activity`
   - `dwd_order_promote_activity`
   - `dwd_order_add_or_sub`
   - `dwd_order_lottery`
   - `dwd_order_vip_award`
6. 跑 11 个 SQL 模板做回归验证

## 8. 每个 SQL 模板最少要验证什么

| 模板 | 最少验证点 |
| --- | --- |
| T01 | 登录/注册/存款/提款/投注/洗码/任务/活动/优惠/彩票/VIP 奖励是否都能汇总到日级别 |
| T02 | 主渠道能拿到折扣，对照渠道无配置时默认 100% |
| T03 | `times = 1` 的首存用户是否正确抽出；新客/老客是否能区分 |
| T04 | `D1~D7` 累计渠道收入是否随时间递增/递减合理；成本项是否正确抵扣 |
| T06 | TOP3 用户是否稳定落到 `P01/P02/P03` |
| T08 | 2充~6充人数与金额是否对应到 `P01/P02/P03` 的充值轨迹 |
| T09 | `ALL / TOPN / NON_TOPN` 三种区间汇总结果是否可区分 |
| T10 | 首存 cohort 的 `D1~D7` 投充比/杀率趋势是否正确 |
| T11 | 游戏类型的有效投注、均注金额、投注占比是否正确 |
| T12 | TOP3 与非TOP3 在游戏类型分布上是否明显不同 |
| T13 | 10 / 20 / 50 / 2000 / 其他 档位人数与占比是否正确 |

---

## 9. 推荐的验收方式

建议每次造完数据后，按下面顺序验收：

1. 先跑 `T02 / T03 / T06 / T13`
   - 这些最容易人工验算；
2. 再跑 `T01 / T08 / T09 / T11 / T12`
   - 校验主统计逻辑；
3. 最后跑 `T04 / T10`
   - 这两个最依赖时间轴、累计口径；
4. 若结果不对，优先排查：
   - 状态码是否一致；
   - 时间字段是否用对（`callback_time / receive_time / send_time / settle_time`）；
   - 渠道过滤是否混入对照渠道数据。
